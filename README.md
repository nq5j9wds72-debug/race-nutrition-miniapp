# Race Nutrition Mini App

Telegram Mini App для расчёта питания на гонку.

Текущий статус проекта: **stable MVP v1.1** после завершения **Дня 29**.  
Это уже рабочий MVP не только с расчётом, но и с реальной памятью продукта: подтверждённые Telegram-пользователи сохраняются в Postgres, а успешные расчёты записываются в историю.

---

# 1. Что делает проект

Приложение помогает получить ориентировочный план на гонку по:

- углеводам
- жидкости
- натрию
- эквиваленту в гелях

Сейчас это уже не просто одноразовый калькулятор, а Telegram Mini App с серверной проверкой Telegram auth, backend-расчётом и сохранением успешных расчётов в Render Postgres.

# 9. Почему архитектура меняется сейчас

## 9.1. Почему больше не SQLite

SQLite была бы временной ступенькой. Для этого проекта это почти наверняка привело бы к повторной миграции позже.

## 9.2. Почему Render Postgres

Потому что проекту уже нужна не просто форма с расчётом, а:

- постоянная память о пользователе
- история расчётов
- нормальный фундамент для роста
- понятная архитектура для демонстрации продукта

## 9.3. Что это даёт владельцу продукта

- не теряется история использования
- можно анализировать реальные сценарии
- можно показывать продукт не как “одноразовое демо”, а как систему
- легче строить следующую продуктовую фазу

## 9.4. Что это даёт бренду / партнёру

- можно показать продукт с памятью, а не просто форму
- можно брендировать UI и слой представления без переписывания расчётного ядра
- позже можно показывать агрегированное использование

## 9.5. Что это даёт конечному пользователю

- не нужно каждый раз начинать с нуля
- появятся последние расчёты
- продукт будет ощущаться более цельным и надёжным

# 10. Что реально считает backend сейчас

Сейчас `server.js` по факту считает так:

- **carbs** = от `duration_min + gi_tolerance_level + effort_level + elevation_gain_m`
- **fluid** = от `sweat_rate_lph`, а если его нет — от `temperature_c` с мягкой поправкой по `humidity_pct`
- **sodium** = от `temperature_c`, рассчитанной жидкости и `sodium_loss_profile`
- **gel equivalent** = от уже рассчитанных углеводов

# 11. Что уже влияет на расчёт, а что пока нет

## 11.1. Поля, которые реально влияют на maths или warnings

Эти поля уже не декоративные:

- `effort_level` → влияет на carbs
- `humidity_pct` → влияет на fluid fallback
- `distance_km` → влияет на sanity-check warnings
- `race_type` → участвует в части warnings и в логике elevation-модификатора
- `elevation_gain_m` → влияет на carbs для `trail / ultra`
- `sodium_loss_profile` → влияет на sodium или warnings

## 11.2. Поля, которые есть в форме, но пока не являются прямыми множителями основной модели

Пока нельзя обещать пользователю как часть “умной модели”, что напрямую в formulas участвуют:

- `weight_kg`
- `distance_km` как прямой множитель formulas
- `race_type` как прямой множитель formulas
- `fuel_format` как прямой множитель formulas

То есть:

- `distance_km` уже влияет на warnings, но не является прямым множителем основной модели
- `race_type` участвует в warnings и в логике elevation-модификатора, но не как отдельный множитель основной модели
- `fuel_format` влияет на warnings, но не меняет formulas
- `weight_kg` пока обязательное поле, но в maths `v1.1` фактически не участвует

# 14. Текущая логика расчёта v1.1

## 14.1. Carbs base

### Если длительность < 60 мин

- `low` = `0 г/ч`
- `medium` = `15 г/ч`
- `high` = `30 г/ч`

### Если длительность 60–150 мин

- `low` = `30 г/ч`
- `medium` = `45 г/ч`
- `high` = `60 г/ч`

### Если длительность > 150 мин

- `low` = `60 г/ч`
- `medium` = `75 г/ч`
- `high` = `90 г/ч`

## 14.2. Effort modifier

- `null` = `0`
- `easy` = `-15 г/ч`
- `steady` = `0`
- `race` = `+15 г/ч`

## 14.3. Elevation modifier

Работает только для `trail / ultra`:

- `< 500 м` = `+0 г/ч`
- `500–1499 м` = `+5 г/ч`
- `1500+ м` = `+10 г/ч`

## 14.4. Carbs clamp

- минимум `0 г/ч`
- максимум `90 г/ч`

## 14.5. Carb interval

- по умолчанию `30 мин`
- если `>45` и `<=75 г/ч` → `20 мин`
- если `>75 г/ч` → `15 мин`

## 14.6. Fluid

Если задан `sweat_rate_lph`:

- `fluid_per_hour_ml = sweat_rate_lph * 1000 * 0.7`

Если `sweat_rate_lph` не задан, используется fallback по температуре:

- `<10°C` → `400 мл/ч`
- `10–19°C` → `500 мл/ч`
- `20–29°C` → `650 мл/ч`
- `>=30°C` → `800 мл/ч`

Потом, только если `sweat_rate_lph` не задан и задан `humidity_pct`:

- `humidity_pct >= 80` → `*1.10`
- `humidity_pct >= 60` → `*1.05`
- `humidity_pct <= 30` → `*0.95`
- иначе → без изменений

## 14.7. Sodium concentration

База по температуре:

- `>=30°C` → `900 мг/л`
- `20–29°C` → `700 мг/л`
- `<20°C` → `500 мг/л`

Потом применяется модификатор `sodium_loss_profile`:

- `low` → `-150 мг/л`
- `medium` → `0`
- `high` → `+150 мг/л`
- `unknown` → `0`
- `null` → `0`

После этого действует clamp:

- минимум `300 мг/л`
- максимум `1100 мг/л`

## 14.8. Интервалы

- fluid interval = `15 мин`
- sodium interval = `15 мин`

## 14.9. Gel basis

- `1 гель = 25 г углеводов`

# 16. Что подтверждено по warnings

## 16.1. Sanity-check warnings

Если `avgSpeedKmh < 2`:

- `Проверь дистанцию и длительность: средняя скорость получилась слишком низкой.`

Если `avgSpeedKmh > 25`:

- `Проверь дистанцию и длительность: средняя скорость получилась слишком высокой.`

Если:

- `race_type = road`
- `distance_km <= 21.1`
- `duration_min >= 300`

то добавляется warning:

- `Проверь дистанцию, длительность и тип гонки: для road такой сценарий выглядит необычно.`

## 16.2. Practical warnings

Если `carbs_per_hour >= 75`:

- `Высокий план по углеводам лучше заранее протестировать на тренировке.`

Если `fuel_format = drink_only` и `carbs_per_hour > 60`:

- `Только напитком такой объём углеводов набрать может быть неудобно.`

Если `sweat_rate_lph` не задан и `temperature_c >= 20`:

- `В жару без данных о вашей потливости точность расчёта жидкости ниже.`

Если `duration_min > 720`:

- `Очень длинная гонка: расчёт носит ориентировочный характер и требует проверки на практике.`

Если `sodium_loss_profile = unknown`:

- `Профиль потерь натрия не указан точно: план по натрию лучше проверить на тренировке.`

Всегда добавляется:

- `Натрий — это ориентир, а не защита от перепивания.`

# 17. Безопасность

Для этого проекта базовые правила такие:

- нельзя доверять данным из клиента
- решения по auth принимаются только после серверной проверки `initData`
- `initDataUnsafe` можно использовать только для UI, но не для доступа, ролей, денег и лимитов
- секреты хранятся только на сервере в env
- продуктовая логика должна оставаться на сервере, а не во frontend

# 22. Правила на всю фазу

1. Не менять математику в те же дни, когда внедряется БД.
2. Не смешивать DB schema и большой UI cleanup в одном дне.
3. История расчётов важнее persistent metrics.
4. Если БД подключена, но пользователь не получает от неё пользы — фаза не завершена.
5. Если UI стал красивее, но всё ещё похож на debug-страницу — фаза не завершена.
6. Любое новое поле в БД добавлять только если можно в 1–2 строках объяснить, зачем оно нужно сейчас.
